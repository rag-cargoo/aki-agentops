# 📝 문서 작성 표준 (Documentation Standard)

> **Core Philosophy**: "문서는 단순히 기록하는 것이 아니라, 설계자의 의도를 시각적으로 전달하는 지도(Map)여야 한다."

---

## 1. 시각적 위계 (Visual Hierarchy)

### 1.1. 헤더 구조 (Headers)
*   **H1 (#)**: 프로젝트명 또는 문서 대주제 (문서당 1회)
*   **H2 (##)**: 주요 단계 (Step/Phase) 또는 대분류
*   **H3 (###)**: 세부 구현 내용, 실험 방법 등
*   **H4 (####)**: 코드 설명, 작은 팁 등

### 1.2. 트리 구조 목차 (Tree-style TOC)
문서 상단에는 반드시 아래와 같은 트리 형태의 목차를 배치하여 구조를 명시한다.
```markdown
## 📂 목차 (Table of Contents)
└─ 🚀 Step 1: 낙관적 락
   ├─ 🛠️ 구현 방법
   └─ 🧪 실험 결과
└─ 🚀 Step 2: 비관적 락
```

---

## 2. 이모티콘 사용 가이드 (Emoji System)

이모티콘은 가독성을 돕는 '이정표'로만 사용하며, 남용을 금지한다.

| 이모티콘 | 용도 (Usage) |
| :--- | :--- |
| 🚀 | 단계(Step) 또는 새로운 페이즈의 시작 |
| 💡 | 핵심 개념, 이론, 아키텍처 결정 사항(ADR) |
| 🛠️ | 실제 코드 구현, 설정, 도구 사용 |
| 🧪 | 실험 시나리오, 테스트 케이스 |
| ✅ | 성공, 완료, 검증된 결과 |
| ❌ | 실패, 오류 상황, 수정이 필요한 레거시 |
| ⚠️ | 주의 사항, 보안 위험, 성능 병목 |

---

## 3. 단락 및 가독성 (Readability)

*   **수평선 (`---`)**: 주요 H2 섹션이 끝날 때마다 수평선을 사용하여 시각적으로 단락을 완전히 분리한다.
*   **여백**: 헤더 위아래로 반드시 한 줄 이상의 공백을 두어 텍스트가 뭉쳐 보이지 않게 한다.
*   **강조**: 중요한 키워드는 `**굵게**` 표시하고, 코드는 `코드 블록` 또는 `inline code`를 사용한다.
