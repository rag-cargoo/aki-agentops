# 문서 작성 표준 (Documentation Standard) - v2.0

> **Core Philosophy**: "장식보다는 위계를, 화려함보다는 여백을 우선하여 설계의 구조가 한눈에 들어오게 한다."

---

## 1. 헤더 및 본문 위계 (Hierarchy)

### 1.1. 단계별 구분 (H2)
- 모든 주요 단계(Step)는 `##` 헤더를 사용하며, 위아래로 2줄 이상의 공백을 둔다.
- 단계 사이에는 반드시 수평선(`---`)을 삽입하여 단락을 분리한다.

### 1.2. 본문 테두리 및 들여쓰기 (Boxing & Indentation)
- **박스 효과**: 각 단계의 상세 내용은 마크다운 인용구(`> `)를 사용하여 전체를 감싼다. 이를 통해 시각적인 "테두리" 안에서 내용이 전개되는 효과를 준다.
- **들여쓰기**: 세부 항목은 반드시 리스트(`-`)를 사용하며, 하위 항목으로 갈수록 2칸씩 들여쓰기를 적용한다.

---

## 2. 표기 원칙 (Notation)

### 2.1. 이모티콘 사용 제한
- 헤더와 본문에서 불필요한 이모티콘 사용을 전면 금지한다. (가독성 저해 방지)
- 상태 표시(성공/실패) 등 핵심적인 지표에만 제한적으로 사용한다.

### 2.2. 코드 및 강조
- 실제 소스 코드는 Fenced Code Block(```)을 사용하여 본문과 명확히 격리한다.
- 중요한 키워드만 **굵게** 표기한다.

---

## 3. 템플릿 예시 (Sample)

## Step X: 제목 (Header)
---
> ### 1. 서론 (Sub-header)
> - 주요 내용 설명입니다.
>   - 하위 세부 내용입니다. (들여쓰기 적용)
>
> ### 2. 구현 코드 (Code Box)
> ```java
> public void logic() { ... }
> ```
>
> ### 3. 검증 결과 및 효과 (Results & Impact)
> - **실제 출력 로그**: 테스트 스크립트 실행 시의 실제 JSON 또는 콘솔 출력값 박스.
> - **비포/애프터 비교**: 이전 단계 대비 기술적/비즈니스적 개선 사항 명시.
> - **적용 효과**: 부하 감소율, 응답 시간 개선 등 구체적 지표.
