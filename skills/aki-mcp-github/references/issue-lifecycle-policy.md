# Issue Lifecycle Policy (Reopen-First)

## 목적
- 이슈 난발을 방지하고, 같은 작업 축은 기존 이슈 갱신/재오픈으로 추적 일관성을 유지한다.

## 기본 원칙
1. 같은 범위/목적의 후속 작업은 새 이슈를 만들지 않는다.
2. 기존 이슈 본문은 보존하고, 변경사항은 코멘트/체크리스트로 누적한다.
3. 새 이슈는 범위가 달라졌을 때만 생성한다.
4. 새 이슈를 만들면 관련 기존 이슈를 반드시 링크한다.

## 결정표 (Reopen vs New)
1. Reopen/Update Existing:
   - 원래 완료기준 내 미완료/회귀/보강 작업
   - 동일 기능 축/동일 산출물 보정
2. New Issue:
   - 목표/범위/종료판정이 기존 이슈와 다름
   - 담당 팀/우선순위/배포 시점이 분리됨
   - 리스크 성격이 달라 독립 추적이 필요함

## 필수 절차
1. 중복 탐색:
   - 키워드로 open/closed 이슈를 검색한다.
2. 기존 이슈 우선 처리:
   - open이면 코멘트로 진행 단계 누적
   - closed이면 reopen 후 코멘트로 진행 단계 누적
3. 새 이슈 생성은 예외:
   - "왜 기존 이슈 재오픈이 아닌지" 근거를 본문/PR에 명시
4. 종료 시점:
   - 완료 근거(검증 로그/링크) 코멘트 후 close

## 진행로그 형식
1. Planning
2. Implementation
3. Verification
4. Closeout

## 자동화 권장
- 로컬 실행은 `scripts/issue-upsert.sh`를 우선 사용한다.
- PR 본문은 저장소 템플릿을 사용하고, `pr-issue-governance` 워크플로우 검증을 통과해야 한다.
