## 목표
- 사용자가 작업 흐름을 즉시 파악할 수 있도록 시작-진행-완료를 체크리스트로 고정한다.
- 코드 수정, 검증, 배포 단계의 누락을 줄인다.
- 작업 중간 상태를 짧고 일관된 포맷으로 보여준다.
- 필요 시 진행상태를 GitHub issue 코멘트로 동기화한다.

## 오케스트레이션 경계
- 이 스킬이 담당:
  - 체크리스트 기반 실행 가시화
  - 상태 전이 규칙(`pending -> doing -> done/blocked`)
  - 완료 보고 포맷(검증/증빙 포함)
  - 이슈 코멘트 동기화 자동화 스크립트 제공
- 이 스킬이 직접 담당하지 않음:
  - 도메인별 구현 로직(백엔드/프론트 비즈니스 규칙)
  - 세션 리로드/브랜치 정책 자체

## 실행 계약
1. 시작 체크리스트를 먼저 제시한다.
   - 본격 탐색/수정 전에 4~8개 항목으로 분해한다.
   - 항목은 동사로 시작하고 결과물이 보이게 작성한다.
2. 진행 중 상태를 계속 갱신한다.
   - 큰 단계가 바뀔 때마다 `[~]` 항목과 새 정보 1~2문장을 공유한다.
   - 막힘이 있으면 즉시 `[!]`로 표시하고 원인을 명시한다.
3. 수정 직전 편집 의도를 먼저 알린다.
   - 어떤 파일을 왜 바꾸는지 한 줄로 설명한다.
4. 검증 단계를 체크리스트에 포함한다.
   - 가능하면 `lint`, `typecheck`, `build`, 핵심 런타임 검증을 모두 실행한다.
   - 미실행 항목은 이유를 기록한다.
5. 완료 보고는 체크리스트 결과와 증빙을 함께 제출한다.
   - 핵심 변경점, 검증 결과, 남은 리스크를 구분해 보고한다.

## 상태 표준
- `[ ]` pending: 아직 시작하지 않음
- `[~]` doing: 현재 진행 중
- `[x]` done: 완료 + 확인됨
- `[!]` blocked: 진행 불가(원인/필요조건 필수)

## GitHub 이슈 동기화
- 실행 스크립트:
  - `scripts/checklist-issue-sync.sh`
- 목적:
  - 체크리스트/검증/증빙을 하나의 코멘트로 업로드하거나(upsert), 새 코멘트로 누적(append)한다.
- 기본 모드:
  - `upsert`(같은 marker를 가진 내 코멘트가 있으면 수정, 없으면 신규 생성)
- 입력:
  - `--issue`, `--checklist-file` (필수)
  - `--repo`, `--summary`, `--verification-file`, `--evidence-file` (선택)

## 완료 정의
- 아래 조건을 모두 만족하면 완료다.
  1. 시작 체크리스트 제시
  2. 중간 상태 갱신 수행
  3. 검증 단계 실행 또는 미실행 사유 명시
  4. 최종 결과 보고(변경/검증/리스크)
  5. (요청 시) GitHub issue 코멘트 동기화

## 결과 보고
- 최종 보고는 아래 순서를 따른다.
  1. 해결 결과 요약(무엇이 달라졌는지)
  2. 체크리스트 최종 상태
  3. 검증 결과(성공/실패/미실행 사유)
  4. 파일/커밋/배포 증빙
  5. (이슈 동기화 시) issue 링크/코멘트 반영 결과
