# Meeting Notes: Infra Approach Decision

## 안건 1: AWS + Terraform + Jump Server를 기본 구축 방식으로 채택할지 검토
- Created At: 2026-04-28 00:45:00
- Updated At: 2026-04-28 00:45:00
- Status: DONE
- 검토사항:
  - 과제 원문은 `온프레미스(폐쇄망 = Air-gapped)`와 `인터넷이 불가능한 네트워크 환경`을 직접 요구한다.
  - AWS `private subnet`과 `jump server`로 격리된 실습 환경을 만들 수는 있지만, 그 자체가 `온프레미스` 요구를 충족한다고 보기는 어렵다.
  - `jump server`를 기본 전제로 두면 제출 설명에서 "완전한 폐쇄망" 해석과 충돌할 여지가 있다.
  - 과제의 핵심 증빙은 네트워크/노드 기본 설정, 오프라인 설치 절차, 서비스 배포와 매뉴얼 정리이며, AWS 프로비저닝 자체는 핵심 요구가 아니다.
- 결정사항:
  - 기본 구축 기준은 `VM 기반 온프레미스 시뮬레이션`으로 둔다.
  - `AWS + Terraform + jump server`는 이 과제의 기본 경로로 채택하지 않는다.
  - 로컬 자원 제약 때문에 AWS를 써야 한다면, 그것은 `air-gap simulation` 대체 실습 환경으로만 간주하고 제출물에 과제 원문 대비 차이와 리스크를 별도로 명시한다.
- 판단근거:
  - 과제 적합성을 최우선으로 두면 `온프레미스`와 `인터넷 불가`를 가장 직접적으로 설명할 수 있는 방식이 유리하다.
  - 초기 단계에서 AWS 설계와 Terraform 작성까지 같이 시작하면 과제 핵심인 오프라인 Kubernetes/DB/모니터링 구축보다 주변 인프라 작업이 앞서게 된다.
