## StorageClass Decision
- Current check: `kubectl get storageclass` -> `No resources found`
- Problem: kubeadm은 control plane과 node bootstrap을 담당하고, Calico는 Pod 네트워크를 담당한다. 둘 다 PV를 자동 생성하는 스토리지 프로비저너를 설치하지 않는다.
- Impact: `04`에서 DB/모니터링 서비스가 PVC를 요청하면 기본 StorageClass가 없어서 PVC가 `Pending`에 머물 수 있다.
- Decision: `04` 서비스 배포 전에 `03` 단계에서 기본 StorageClass를 구성한다.
- Primary option: `local-path-provisioner`
- Alternatives: NFS provisioner, Longhorn
- Fallback only: static PV 또는 hostPath
- Verification criteria:
  - `kubectl get storageclass`에서 기본 StorageClass가 보인다.
  - 테스트 PVC가 `Bound` 상태가 된다.
  - 테스트 Pod가 PVC에 파일을 쓰고 읽을 수 있다.
  - `04` 서비스 values/manifest에서 `storageClassName`을 명시하거나 기본 StorageClass를 사용한다.
- Operation runbook: `workspace/infra/airgap/kubernetes/airgap-k8s/manual/03-쿠버네티스-클러스터-구성/storageclass/operations.md`

## Sidecar Draft
- 실제 클러스터 구성 전까지 아래 내용은 제출 매뉴얼에 넣지 않는다.
- 수동 구성과 Ansible 구성은 둘 다 준비하되, 실행 결과가 생긴 방식만 매뉴얼 본문에 반영한다.
- 이 장은 AWS 인스턴스가 아니라 VM 노드 기준 설치 절차로 진행해도 무방하다.
- 사전점검 스크립트: `ops/03-kubernetes-cluster/manual-kubeadm/02-preflight/scripts/02-01-kubeadm-preflight-check.sh`
- 로컬 작업 원본: `assets/offline-assets/`
- 반입 번들 경로: `delivery/offline-assets/`
- 서버 반입 경로: `/opt/offline-assets`
- 서버 설치 명령은 `/opt/offline-assets/...` 기준으로 작성한다.
- bastion 경유 접속 명령은 `manual/03-쿠버네티스-클러스터-구성/README.md` 기준으로 기록한다.
- `nc` 포트 점검 예시는 kubeadm 설치 전 추가 검증으로 `manual/03`에 둔다.
- `03` 구조는 `manual-kubeadm/`, `ansible-kubeadm/`, `calico/`로 분리하고, bastion/control node/IP 변경은 `.env`와 각 설치 방식의 bootstrap에서 흡수한다.
- 수동은 `01`~`10` 단계 번호로 실행하고, Ansible은 같은 목적의 자동화 단계를 별도 디렉터리에서 관리한다.
- 실제 실행 단계는 `step -> verify -> state marker` 구조로 맞춘다. `all-verify`는 준비 스크립트 문법이 아니라 실제 완료 마커와 독립 verify 결과를 읽는다.
- `03-01-preflight-run`, `03-01-preflight-verify`, `03-01-preflight-clear`는 기존 이름 호환용 alias로 유지한다.
- `03-02` 수동 kubeadm은 `manual-kubeadm/01..10/scripts/*.sh`와 `step-03-02-01..10-*` target을 1:1로 맞춘다.
- `04` 서비스 배포 전 PVC 자동 바인딩을 위해 `03` 말미에 StorageClass 구성을 추가한다.
