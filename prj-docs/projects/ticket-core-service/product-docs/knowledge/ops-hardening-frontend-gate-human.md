## 1. 용어 사전
- Runtime Integration Test:
  - Redis/Kafka/Postgres를 실제로 띄운 상태에서 핵심 통합 테스트를 실행하는 검증입니다.
- Release Gate:
  - 머지 전 반드시 통과해야 하는 검증 묶음(compile + 핵심 회귀 테스트)입니다.
- Baseline:
  - 성능의 최소 기준선입니다. 다음 변경에서 이 수치가 악화되는지 비교 기준이 됩니다.
- Observability:
  - 운영 중 문제를 관찰할 수 있게 로그/메트릭/스냅샷을 갖추는 체계입니다.
- Runbook:
  - 장애가 났을 때 즉시 실행할 복구 절차 문서입니다.
- Frontend Gate:
  - 프론트 구현을 시작해도 되는지 판단하는 사전 완료 조건 집합입니다.

## 2. 왜 DDD 다음에 운영 고도화를 먼저 했는가
- DDD/헥사고날 리팩토링은 "코드 경계"를 안정화합니다.
- 프론트 착수는 "운영 안정성"이 확보돼야 개발/테스트/릴리즈가 반복 가능합니다.
- 경계만 정리된 상태에서 프론트를 붙이면, 장애/회귀가 생길 때 원인 분리가 어려워집니다.
- 그래서 이번에는 DDD 이후 바로 운영 고도화 7종을 선행해, 프론트가 안전한 바닥 위에서 시작되도록 했습니다.

## 3. 이번 7개 항목을 한 번에 처리한 이유
- 항목이 서로 연결되어 있기 때문입니다.
  - 통합테스트 없이는 운영 헬스 판정이 공허합니다.
  - 성능 기준선 없이는 릴리즈 판단이 주관적이 됩니다.
  - 관측/런북이 없으면 장애가 나도 복구 시간이 길어집니다.
- 한 번에 묶어 완료하면 "프론트 시작 가능/불가"를 명확히 말할 수 있습니다.

## 4. 패키지/문서 구조를 이렇게 둔 이유
- 스크립트는 실행 목적별로 분리했습니다.
  - `scripts/ops/*`: 운영 점검/게이트/통합 런
  - `scripts/perf/*`: 부하/기준선
- 문서는 역할별로 분리했습니다.
  - `api-specs/*`: 운영 기준(관측/장애 대응)
  - `api-test/*`: 실행 절차와 결과 해석(통합/성능/릴리즈 게이트)
  - `knowledge/*`: 왜 이렇게 하는지, 사람이 이해하기 위한 배경 설명
- sidecar(`prj-docs`)를 문서 SoT로 두고, 제품 레포는 실행 자산(코드/스크립트/CI)에 집중합니다.

## 5. 프론트 전환 게이트(실행 기준)
- 아래 5개를 모두 만족하면 프론트 구현 페이즈로 전환합니다.
  - 1) Runtime Integration Report: `PASS`
  - 2) Release Gate Report: `PASS`
  - 3) k6 Baseline + Distributed Report: `PASS`
  - 4) 관측/장애 대응/릴리즈 가이드 문서 최신화 완료
  - 5) `task.md`의 `TCS-SC-027` 상태 `DONE`
- 기준 미충족 시 프론트 구현을 시작하지 않고, 해당 항목을 먼저 복구합니다.
